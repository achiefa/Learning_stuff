Detailed Description

Generate and resolve graph of dependencies.

The nodes in this graph are the instance factories. There is an edge between nodes A and B iff the instance factory B requires the service produced by the instance factory A. The order in which services are added to the injector is determined by doing a topological sort of the dependency graph.

Register a new instance factory.

Instance factories can be registered in arbitrary order as no instances are actually created at the time of registration.

instance_factory
  1. Locate the graph node corresponding to the InstanceType (the type of the service we want to use). If that node doesn't exist yet, it will be created.
  2. Assign an initializer function to the node. The initializer function is any callable that creates the service (passing its dependencies through the corresponding factory instance_factory) and then places it into the given injector. Note that, in order for inj.inject() to pass all the needed dependencies to the instance factory, these dependencies must have been added previously to the graph.
  3. For each dependence in the variadic template parameter Deps, an edge is added between the new node and the node in the graph corresponding to the dependency. The function passthru is needed in order to expand the parameter pack, thus using each parameter in it.

injector di_config::build_injector ( )

Create a new injector with services created by the registered instance factories.

Build the injector.

This method is where the dependencies are actually resolved and passed to factories. This method also ensures that instances are created in the correct order.

Returns
injector
  1. Determine the correct order for calling the initializers by using the topological sort on the graph.
  2. Call the initializers in that order.

The stack initializers is initially empty. It is filled up, according to the topological order, by toposort_visit_node. Then, the stack is exhausted by calling each element.
